# checkmd — 기능 구현 계획 문서 생성/업데이트

> 해당 문서에 메모/주석 달아서 질문한 내용을 보고 Claude가 업데이트합니다.

## 역할

이 스킬은 기능 구현 계획 문서(plan 파일)를 **일관된 포맷**으로 생성하거나 업데이트합니다.

### 호출 방법

```
/checkmd [파일경로]
```

- 파일이 없으면 → 새 계획 문서 생성 (대화 컨텍스트 기반)
- 파일이 있으면 → 파일 내 주석/질문을 읽고 업데이트. **바로 코드 작성 금지**

---

## 동작 규칙

1. **계획 단계**: 사용자가 "구현하자"라고 하기 전까지 문서만 업데이트, **바로 코드 작성 금지**
2. **질문 처리**: 문서 내 사용자 주석(`> 질문`)을 Q&A 형식으로 변환하여 논의 섹션에 추가
3. **맥락 관리**: 질문내용을 기반으로 논의 섹션에 추가해서 어떤 맥락으로 이런 결정이 나왔는지 추가
4. **버전 관리**: 모든 변경은 변경 이력에 버전 태그(`v1.0`, `v2.0`, ...)와 **날짜를 반드시 기록**
5. **구현 후**: 완료된 Step은 `[x]` 체크, 변경 파일 목록에 추가, 다음 단계 명시
6. **마지막 체크**: 구현 후 또는 수정 완료 후 **반드시** 간단한 작업은 결과를 직접 실행해보고, 복잡한 작업이라면 브라우저를 직접 켜서 잘 작동하는지 확인
6-1. **교훈 반영 (자기개선 루프의 "읽기" 고리)**: 계획을 세울 때 누적 교훈(`~/.claude/harness/lessons.md`, SessionStart 훅으로 컨텍스트에 자동 주입됨)을 참고한다. 과거에 밟은 함정·내린 결정이 이번 계획의 재사용·안전성 게이트와 충돌하지 않는지 확인하고, 관련 교훈은 해당 섹션에 명시적으로 반영한다.

### Plan 비평가 (A — Q&A 루프를 깊게 만드는 안쪽 고리)

6-2. **비평가 역할**: plan을 갱신할 때마다 응답 끝에서 **"아직 안 정해진 것"을 스스로 캐낸다.** 사용자 질문만 기다리지 말고 AI가 먼저 날카로운 질문을 던져 Q&A 루프가 끊기지 않게 한다.
   - **탄약**: 주입된 누적 교훈(lessons)을 근거로, 과거 실수가 이번 plan에 반영됐는지 묻는다 ("지난번 X를 빼먹어 터졌는데, 이 plan엔 반영됐나?").
   - **무엇을 캐나**: 모호한 요구사항, 빠진 엣지케이스, 서로 충돌하는 결정, 비어 있는 안전성·재사용 항목, 검증 안 된 가정.
   - **출력 위치**: 캐낸 질문은 `🔎 비평: 열린 질문` 섹션에 `- [ ]`로 남긴다.
   - **선 넘지 않기**: **스스로 답하지 마라.** 사용자 판단을 기다린다 (반자율 원칙).
   - **해소**: 사용자가 답하면 → 해당 항목을 `[x]`로 바꾸고, 답변과 근거를 `💬 논의 & 결정 사항`으로 옮긴다.
   - **멈춤 조건**: 더 캐낼 게 없으면 "열린 질문 없음 — 구현 준비됨"이라고 명시한다 (B 게이트 통과 신호).

### 코드 구조 게이트 (계획 시 항상 자문)

7. **재사용 게이트**: 새로 짜기 전에 "이미 있는 함수·컴포넌트·유틸로 되나"를 먼저 찾는다.
   중복 구현을 만들지 말고, 기존 패턴/추상화를 재사용한다.
8. **오버엔지니어링 게이트**: 지금 요구사항에 필요한 최소 구현을 한다.
   "나중에 쓸지도 모르니" 추상화·옵션·일반화를 미리 넣지 않는다(YAGNI).
   복잡한 추상화보다 지루하지만 명확한 코드를 우선한다.

### 안전성 게이트 (외부 입력·상태 변경·부수효과·비밀값·비가역 작업이면 자문)

9. **신뢰 경계 게이트**: 시스템 밖에서 들어온 값(클라이언트 입력·URL 파라미터·외부 API 응답·
   웹훅 payload)은 전부 미검증 데이터로 취급한다. "이 값 어디서 왔고 검증됐나"를 항상 자문.
   - 중요한 값(금액·식별자·권한·상태)은 클라이언트가 보낸 걸 믿지 말고 서버의 신뢰 가능한
     소스(DB·인증된 세션·원천 API)로 재검증한다.
   - 외부에서 호출 가능한 엔드포인트(웹훅·콜백)는 공개 표면이다 — 처리 전에
     "진짜 그 발신자가 맞나"(서명·시크릿·토큰)부터 검증한다. (웹훅 자체는 정상 패턴이며 금지가 아니다)
10. **멱등성·동시성 게이트**: 상태를 바꾸는 동작은 "두 번 실행되면? 동시에 실행되면?"을
    먼저 자문한다. 네트워크 재시도·중복 콜백·버튼 연타·동시 요청에도 결과가 한 번만
    반영되도록 설계한다(고유키·상태 가드·잠금). 부수효과(결제·발송·과금)일수록 필수.
11. **실패 경로 게이트**: 모든 I/O(네트워크·DB·파일·외부 API)는 실패한다고 전제한다.
    happy path만 짜지 말고 타임아웃·에러·부분 실패 시 시스템이 어떤 상태로 남는지 계획에 적는다.
    특히 "A는 성공했는데 뒤따르는 B가 실패"한 중간 상태의 복구·재시도 경로를 명시한다.
12. **시크릿·노출 게이트**: 비밀값(키·토큰·자격증명)이 클라이언트 번들·로그·에러 메시지·
    URL·분석 이벤트로 새지 않는지 확인한다. 클라이언트로 내려가는 값과 서버 전용 값의
    경계를 분명히 하고, 개인식별정보(PII)는 꼭 필요한 곳에만 최소로 흐르게 한다.
13. **관측가능성 게이트**: 기능이 실패했을 때 추적 단서가 남는지 확인한다. 무엇이 어디서 왜
    실패했는지 로그·알림으로 알 수 있어야 한다. catch 후 조용히 삼키는 "silent fail" 금지.
14. **되돌릴 수 있나 게이트**: 비가역 작업(스키마 변경·데이터 삭제·대량 수정·외부 발송)은
    실행 전 한 번 더 자문한다. 롤백 경로가 있는가, 먼저 백업/드라이런이 가능한가,
    영향 범위가 예상과 같은가. 되돌리기 어려우면 계획에 별도로 표기한다.
