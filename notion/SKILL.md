# 노션 일일보고서

**어투는 간결하게(~수정, ~함 등), 내용은 최대한 상세하게** 개발 일일보고서를 생성합니다.

이 일지는 나중에 "그때 왜 그렇게 했지?"에 답하기 위한 기록입니다.
읽는 사람이 원본 plan MD를 열어보지 않아도 될 만큼 맥락·근거·대안이 다 들어가야 합니다.
요약본이 아니라 **의사결정 기록**으로 씁니다.

## 실행 방법

- 인자 없이 `/notion` → 오늘 날짜 기준 자동 수집
- `/notion 2026-05-12` → 특정 날짜 지정
- `/notion 2026-05-10 2026-05-12` → 날짜 범위 지정

$ARGUMENTS 가 없으면 오늘 날짜를 사용합니다.

## 데이터 수집 순서

### Step 1: 오늘 커밋 수집

```bash
git log --since="대상날짜 00:00" --until="대상날짜+1 00:00" --oneline --all
```

### Step 2: 오늘 변경된 plan MD 파일 특정

```bash
git log --since="대상날짜 00:00" --until="대상날짜+1 00:00" --all \
  --name-only --pretty=format:"---%h %s" | grep -E "^(---|docs/plans/)"
```

`HEAD~N` 방식은 머지 커밋·다른 브랜치가 섞이면 어긋나므로, 날짜 범위로 직접 뽑습니다.
plan 문서 경로는 프로젝트마다 다릅니다(`docs/plans/`, `docs/plan/`, `.claude/` 등) — 먼저 `ls`로 확인하세요.

### Step 3: 변경된 plan MD에서 오늘 추가된 Q&A 섹션 읽기

```bash
git diff <커밋>~1 <커밋> -- "docs/plans/해당파일.md" | grep "^+"
```

diff에서 `+### Q숫자.` 패턴으로 새로 추가된 Q&A 섹션 번호를 파악한 뒤,
해당 Q&A 섹션 **전체**를 읽습니다 (배경, 비교 분석, 의사결정 테이블, 구현 결과 포함).

또한 `<details><summary>vX.X (오늘날짜)` 블록도 함께 읽어 변경 요약을 확인합니다.

### Step 4: 노션 리포트 작성

수집한 데이터를 아래 형식으로 변환합니다.
Q&A의 고민 과정과 의사결정 이유가 자연스럽게 기술적 내용으로 녹아들도록 작성합니다.

**매핑 규칙:**
- Q&A의 **배경** → 상황(S)
- Q&A의 **과제/요구사항** → 과제(T)
- Q&A의 **Step 체크리스트 + 구현 결과** → 내가 한 일(A)
- Q&A의 **대안 비교 테이블 + 결정 근거** → 기술적 선택 및 이유
- Q&A의 **A안/B안/C안 검토 + 왜 채택/기각** → 설계 고민 및 의사결정 기록
- Q&A의 **개선 효과** → 개선 효과
- 커밋 해시 + 메시지 → 주요 커밋 목록

### Step 5: 깊이 검사 (작성 후 반드시 수행)

초안을 쓴 뒤 아래 기준으로 self-check하고, 미달이면 **원본 Q&A를 다시 읽어 보강한 뒤 업로드**합니다.
"간결한 어투"는 **문체**(~수정, ~함)를 뜻하지, **내용을 줄이라는 뜻이 아닙니다.**

**분량 기준 (섹션당):**

| 섹션 | 최소 분량 | 미달 신호 |
|---|---|---|
| 📌 목적 | S/T 각 2~4문장 | "성능을 개선함" 같은 한 줄 |
| 🛠️ 내가 한 일 | 카테고리 2~5개 × 각 3~6개 항목 | 커밋 메시지를 그대로 나열만 함 |
| 💡 기술적 선택 및 이유 | 결정 1건당 5줄 이상, 결정 2건 이상 | 채택안만 있고 기각안이 없음 |
| 🧩 설계 고민 및 의사결정 기록 | 고민 1건당 문단 하나 | "SRP를 고려해 결정함" 같은 추상어 |
| 🔍 개선 효과 | Before → After 3항목 이상 | 수치·근거 없는 "개선됨" |
| 🤝 협업/영향/재사용 | 3항목 이상 | 전부 "해당 없음" |

**반드시 포함할 구체 정보 — 없으면 원본에서 찾아옵니다:**

1. **실제 파일 경로·함수명·컴포넌트명** (`lib/pearl.ts:syncWallet`) — "지갑 로직"처럼 뭉뚱그리지 않습니다
2. **수치** — 라운드트립 20회 → 1회, 렌더 3.2s → 0.4s, 파일 13개 등. 측정값이 있으면 반드시 인용
3. **기각한 대안과 기각 사유** — 채택안만 적으면 절반짜리 기록입니다. 대안이 2개 이상이면 비교 테이블(항목·A안·B안·결정)로
4. **의견이 바뀐 지점** — "처음엔 X로 갔다가 → Y 때문에 Z로 선회". 이게 일지의 가장 큰 가치이므로 생략 금지
5. **밟은 함정·삽질** — 원인과 해결까지. 성공 경로만 적으면 나중에 같은 함정을 다시 밟습니다
6. **핵심 코드 스니펫** — 설명만으로 부족한 결정은 5~15줄 코드 블록으로. 단, 붙여넣기 덤프는 금지
7. **트레이드오프와 남은 부채** — 이번에 감수한 비용, 다음에 갚아야 할 것

**하지 말아야 할 것:**

- 원본 Q&A에 있는 고민 과정을 한 줄로 압축하는 것 (일지의 존재 이유가 사라짐)
- 근거 없이 분량만 늘리는 것 — 원본에 없는 내용을 **추측해 채우지 않습니다.** 정보가 없으면 `- (기록 없음)`으로 남깁니다
- 각 섹션을 서로 베끼는 것 (기술적 선택 ↔ 설계 고민은 **관점이 다릅니다**: 전자는 "무엇을 골랐나", 후자는 "어떻게 고민했나")

## 리포트 형식

### ⚠️ 형식의 SSOT는 노션 템플릿 페이지("제목")입니다

일지 DB에는 **`제목`이라는 이름의 템플릿 페이지**가 들어 있습니다(각 DB마다 1개, 아이콘 💻).
본문 섹션 구성은 **이 문서가 아니라 그 템플릿이 정답**입니다. 노션에서 템플릿을 고치면 리포트도 따라가야 하므로,
아래에 적힌 형식은 템플릿을 못 읽었을 때의 폴백일 뿐입니다.

**작성 전 반드시 템플릿을 읽습니다:**

1. `mcp__notion__API-post-search`로 `"제목"`을 검색 (filter: `{"property":"object","value":"page"}`)
2. 결과 중 **`parent.database_id`가 대상 일지 DB인 페이지**를 고름 (여러 DB에 동명 템플릿이 있으므로 반드시 걸러야 함)
3. `mcp__notion__API-retrieve-page-markdown`으로 본문을 읽음
4. 거기 나온 **H1 섹션 제목·순서·구분선(`---`)을 그대로 재현**하고, 각 섹션의 안내 문구(예: "어떤 선택지를 두고 고민했는가")를
   이번 작업의 실제 내용으로 치환합니다. 안내 문구 자체를 그대로 남기지 않습니다.
5. 내용이 없는 섹션이라도 **섹션은 지우지 않고**, 해당 없으면 `- 해당 없음`으로 채웁니다 (템플릿 골격 유지)

템플릿 페이지를 못 찾으면 사용자에게 알리고 아래 폴백 형식으로 진행합니다.
**템플릿 페이지 자체는 절대 수정·삭제하지 않습니다** (중복 방지 로직이 이 페이지를 기존 일지로 오인하지 않도록,
제목이 정확히 `제목`인 페이지는 항상 제외).

### 페이지 제목
`[MM/DD] 핵심작업 요약 (브랜치명)`
— 제목만 봐도 뭘 했는지 알 수 있게 구체적으로

### 본문 (폴백 — 템플릿을 못 읽은 경우)

## MD 파일 저장

리포트를 `docs/notion/` 디렉토리에 MD 파일로도 저장합니다.
파일명: `YYYY-MM-DD_핵심키워드.md`

## 노션 업로드

Notion MCP 도구(`mcp__notion__API-post-page`, `mcp__notion__API-patch-block-children`)를 사용합니다.

### 대상 데이터베이스 결정 (프로젝트마다 다름 — 하드코딩 금지)

일지 DB는 **프로젝트마다 다르므로 이 문서에 ID를 고정하지 않습니다.** 아래 순서로 결정합니다:

1. **사용자가 명시한 경우** — `/notion 7월 16일꺼 OO 일지에` 처럼 DB명을 지정하면 그것을 사용
2. **프로젝트 설정 파일** — 해당 프로젝트의 `.claude/notion-db.json`이 있으면 사용
   ```json
   { "database": "<이 프로젝트의 일지 DB 제목>", "database_id": "<Notion DB UUID>" }
   ```
3. **자동 탐색** — `mcp__notion__API-post-search`로 `"일지"`를 검색해 DB 목록을 얻고, 프로젝트명(디렉토리명·`package.json` name·브랜드 상수 파일 등)과 가장 잘 맞는 것을 고름
4. **후보가 여러 개거나 확신이 없으면 → 사용자에게 물어봅니다.** 임의로 고르지 않습니다

DB를 새로 확정했으면 `.claude/notion-db.json`에 저장해 다음 실행부터 자동 해석되게 합니다.

> ⚠️ 서비스명이 바뀌어도 노션 DB 제목은 옛 이름으로 남아 있을 수 있습니다(예: 코드상 브랜드는 "몽현"인데 DB는 "몽현당 일지"). 이름이 정확히 안 맞아도 같은 프로젝트일 수 있으니 후보로 고려하되, 애매하면 물어봅니다.

### 중복 방지 (업데이트 모드)

노션에 작성하기 전 반드시 기존 페이지 존재 여부를 확인합니다:

1. `mcp__notion__API-post-search`로 대상 날짜 제목(`[MM/DD]`)을 검색하고, 결과 중 **parent.database_id가 대상 DB인 것**만 추립니다
   (`API-post-database-query`는 존재하지 않는 도구이고, `API-query-data-source`에 database_id를 넣으면 400이 납니다)
   - 제목이 정확히 `제목`인 페이지는 **템플릿이므로 후보에서 제외**합니다
2. **기존 페이지가 있으면:**
   - 해당 페이지의 기존 블록을 `mcp__notion__API-get-block-children`으로 조회
   - 기존 블록을 모두 `mcp__notion__API-delete-a-block`으로 삭제
   - 새 내용을 `mcp__notion__API-patch-block-children`으로 추가
   - 제목이 변경되었으면 `mcp__notion__API-patch-page`로 제목도 업데이트
3. **기존 페이지가 없으면:** 새 페이지 생성 (`mcp__notion__API-post-page`)

이렇게 하면 `/notion`을 몇 번 실행하든 같은 날짜 페이지는 항상 1개만 유지됩니다.

### 페이지 속성

DB 스키마를 `mcp__notion__API-retrieve-a-database`로 확인해 실제 속성명에 맞춰 채웁니다(프로젝트마다 다를 수 있음).
같은 DB의 최근 페이지 1건을 참고하면 채우는 관례를 알 수 있습니다 — 예를 들어 `날짜`는 페이지 작성일이 아니라 **작업한 날**입니다.

### MD 파일도 동일

`docs/notion/YYYY-MM-DD_*.md` 파일이 이미 있으면 덮어씁니다 (새로 만들지 않음).

## 주의사항

- plan MD가 수십 개·수만 줄인 프로젝트가 많으므로 **절대 전부 읽지 않습니다**
- 오늘 변경된 파일의, 오늘 추가된 섹션만 읽습니다
- Q&A 본문의 고민 과정이 핵심 — changelog 요약만으로는 부족합니다
- **읽는 건 좁게, 쓰는 건 깊게** — 오늘 바뀐 섹션만 읽되, 읽은 건 압축하지 말고 다 풀어씁니다
- 업로드 전 **Step 5 깊이 검사**를 통과해야 합니다 (분량 미달이면 원본 재독 후 보강)
- 같은 날짜 노션 페이지는 중복 생성하지 않고 기존 페이지를 업데이트합니다
- **본문 골격은 항상 대상 DB의 `제목` 템플릿 페이지를 읽어 따릅니다** — 이 문서의 형식은 폴백일 뿐입니다
- `제목` 템플릿 페이지는 읽기 전용 — 수정·삭제하지 않고, 중복 검사 후보에서도 제외합니다
- **DB ID·경로·파일 개수 등 프로젝트마다 달라지는 값을 이 문서에 하드코딩하지 않습니다** — 매번 탐색하거나 프로젝트 설정 파일에서 읽습니다

# 🔁 교훈 증류 (자기개선 루프의 "쓰기" 고리)

보고서 작성을 마친 뒤, **이번 작업에서 다음에 재사용할 교훈을 증류**해 전역 교훈 저장소에 누적한다. 이게 하네스가 복리로 발전하는 엔진이다 (다음 세션 시작 시 SessionStart 훅이 이 파일을 자동 주입한다).

대상 파일: `~/.claude/harness/lessons.md`

규칙:

1. **소스**: 이번 작업의 트러블슈팅(밟은 함정), 설계 의사결정(왜 그 선택), 발견한 프로젝트 특이사항에서 뽑는다.
2. **무엇을 적나**: "다음에 같은 실수를 막아줄 구체적 규칙"만. 한 번성 사건 보고가 아니라 **재발 방지 규칙**.
3. **형식**: 적절한 섹션 아래에 `- [YYYY-MM-DD] 명령형 규칙 — (왜: 한 줄 근거)` 한 줄로.
4. **중복 금지**: 이미 있는 교훈이면 새로 추가하지 말고 기존 항목을 보강한다. 추가 전 파일을 읽고 확인한다.
5. **가볍게**: 뽑을 만한 게 없으면 아무것도 추가하지 않는다. 추상적 다짐("꼼꼼히 하자")은 적지 않는다.
